# Email Verification

## Overview

Email verification confirms that a user owns the email address they registered with. It is implemented as a single-use token sent by email. Verification is not mandatory to use the platform but unlocks the "Email Verified" badge and is a prerequisite for identity verification.

---

## Verification Flow

```
1. User registers
   │
   ├─ API generates: 64 random bytes → base64url encoded = rawToken
   ├─ API stores:    SHA256(rawToken) in EmailVerificationTokens table
   │                 with UserId, ExpiresAt, UsedAt = null
   └─ API sends email (or logs to console in dev):
      Subject: "Verify your email"
      Link: http://localhost:3000/verify-email?token={rawToken}

2. User clicks the link
   │
   Frontend extracts token from query string
   Frontend calls: GET /api/auth/verify-email?token={rawToken}
   │
   ├─ API hashes: SHA256(rawToken)
   ├─ API finds: EmailVerificationTokens row by TokenHash
   ├─ API checks: UsedAt == null (not already used)
   ├─ API checks: ExpiresAt > now (not expired)
   ├─ API sets:   token.UsedAt = now
   ├─ API sets:   user.IsEmailVerified = true
   ├─ API sets:   ProfileIndexes.IsEmailVerified = true
   └─ Returns: 200 OK

3. Frontend shows: "Email verified! You can now close this tab."
```

---

## Resending the Verification Email

If the user loses or never received the email:

```
POST /api/auth/resend-verification
Authorization: Bearer {accessToken}
```

- Invalidates any existing unused tokens for this user
- Generates a new token
- Sends a new verification email (or logs to console in dev)
- Returns 200 OK

Rate-limited along with other auth endpoints (10 req/min per IP).

## Development Behavior

In the development environment, the `DevEmailSender` is registered:

```csharp
builder.Services.AddScoped<IEmailSender, DevEmailSender>();
```

It writes the email content to the API console output instead of sending a real email. Look for output like:

```
[DEV EMAIL]
To: user@example.com
Subject: Verify your MatrimonialBD email
Body:
  Hello user@example.com,
  Please verify your email by clicking the link below:
  http://localhost:3000/verify-email?token=abc123def456...
```

To verify, copy the token value and visit the URL or call the API directly:
```
GET http://localhost:5000/api/auth/verify-email?token=abc123def456...
```

## Production Email

In production, replace `DevEmailSender` with a real implementation. The `IEmailSender` interface:

```csharp
public interface IEmailSender {
    Task SendAsync(string to, string subject, string body, CancellationToken ct = default);
}
```

Recommended providers: **SendGrid**, **AWS SES**, **Postmark**, **Mailgun**.

Example registration in `Program.cs`:
```csharp
if (app.Environment.IsProduction()) {
    builder.Services.AddScoped<IEmailSender, SendGridEmailSender>();
} else {
    builder.Services.AddScoped<IEmailSender, DevEmailSender>();
}
```

Add the provider API key to `appsettings.Production.json` or environment variables.

## Security Design

Tokens are never stored in plaintext. The flow is:

```
Raw token (generated by API)  →  base64url encoded  →  sent in email
                                                                │
                                                    stored in URL only

SHA256(raw token)  →  stored in DB  →  used for lookup
```

If the database is compromised, an attacker cannot forge verification links from the stored token hashes because SHA256 is one-way. The raw token is only ever in the email link.

Token expiry defaults to 24 hours. Expired tokens are rejected but not automatically cleaned up (no background job in v1). Old expired tokens accumulate in the `EmailVerificationTokens` table.

## Effect on Profile

`IsEmailVerified` is stored in two places:
- `Users.IsEmailVerified` (bool) — the authoritative source
- `ProfileIndexes.IsEmailVerified` (bool) — denormalized for display in search

The badge appears on:
- Search result profile cards
- Full profile view
- The user's own profile setup page

---

## Checking Verification Status

The current user's email verification status is returned in:
- `GET /api/auth/me` → `{ isEmailVerified: bool }`
- Stored in `AuthContext.isEmailVerified` on the frontend
- Any page can check `isEmailVerified` to show or hide the verification banner
